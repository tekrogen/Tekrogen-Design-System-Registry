# Install & Configure the shadcn Registry

How to stand up the Tekrogen Design System as a **shadcn registry**: a set of
JSON files that the `shadcn` CLI can install into any consuming project with
`shadcn add @tekrogen/<item>`. Mirrors the [vercel/registry-starter](https://github.com/vercel/registry-starter)
fork goal in the README.

> **pnpm only.** This repo pins pnpm via `devEngines.packageManager`, so every
> command uses `pnpm dlx shadcn@latest …`. **Do not use `npx`** — npm aborts with
> `EBADDEVENGINES` here (same reason the husky hook uses `pnpm exec`).

---

## Mental model: two halves

A registry has an **authoring** side (this repo) and a **consuming** side (any
app that installs from it):

```
authoring (here)                         consuming (any project)
─────────────────                        ───────────────────────
registry/<style>/<item>/*.tsx            components.json → registries: { "@tekrogen": ".../r/{name}.json" }
registry.json  (manifest)                       │
   │  pnpm registry:build                       ▼
   ▼                                      pnpm dlx shadcn@latest add @tekrogen/<item>
public/r/<item>.json  (built, served)  ──────►  files copied into the consumer's project
```

You **author** source files + a manifest, **build** them into static JSON under
`public/r/`, **host** that (Vercel), and consumers **install** items by namespace.

---

## Prerequisites

- pnpm `^11.9.0` (enforced; see `package.json` `devEngines`).
- The build script already exists in `package.json`:
  ```json
  "scripts": { "registry:build": "pnpm dlx shadcn@latest build" }
  ```
- Tailwind CSS + a `components.json` (created in Step 1). The registry-starter
  fork supplies the Next.js app that serves `public/r/` — if you haven't pulled
  it in yet, the registry JSON can still be built and hosted as static files.

---

## 1. Project config — `components.json`

`shadcn` reads `components.json` at the repo root. Create it (or run
`pnpm dlx shadcn@latest init` to generate one), then add the **`registries`**
block so this project can also *consume* namespaced registries:

```json
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "new-york",
  "rsc": true,
  "tsx": true,
  "tailwind": {
    "config": "",
    "css": "app/globals.css",
    "baseColor": "neutral",
    "cssVariables": true
  },
  "aliases": {
    "components": "@/components",
    "ui": "@/components/ui",
    "lib": "@/lib",
    "hooks": "@/hooks"
  },
  "registries": {
    "@tekrogen": "https://<your-domain>/r/{name}.json"
  }
}
```

- `style` (`new-york` or `default`) sets the shadcn primitive style. **Note:**
  the generic shadcn convention nests source under `registry/<style>/`, but the
  registry-starter base this repo forks organizes by **category** instead —
  `registry/common/`, `registry/layouts/`, etc. Follow the category layout here.
- `registries` maps a **namespace** (`@tekrogen`) to a URL template. `{name}` is
  substituted with the item name at install time. Local dev points at
  `http://localhost:3000/r/{name}.json`.

## 2. Source layout + the manifest — `registry.json`

Put each item's source under `registry/<style>/<item>/`, then describe every item
in a root `registry.json`:

```json
{
  "$schema": "https://ui.shadcn.com/schema/registry.json",
  "name": "tekrogen",
  "homepage": "https://github.com/tekrogen/Tekrogen-Design-System-Registry",
  "items": [
    {
      "name": "login-form",
      "type": "registry:component",
      "title": "Login Form",
      "description": "A login form component.",
      "files": [
        {
          "path": "registry/new-york/login-form/login-form.tsx",
          "type": "registry:component"
        }
      ]
    }
  ]
}
```

`name` is the registry's id; `items[]` is the list the build compiles. Each entry
is a **registry item** (full field reference below).

## 3. Anatomy of a registry item

A complete item with dependencies, multiple files, and theme tokens:

```json
{
  "name": "hello-world",
  "type": "registry:block",
  "title": "Hello World",
  "description": "A simple hello world block.",
  "dependencies": ["zod", "motion"],
  "devDependencies": ["tw-animate-css"],
  "registryDependencies": ["button", "input", "@tekrogen/brand-tokens"],
  "files": [
    { "path": "registry/new-york/hello-world/hello-world.tsx", "type": "registry:component" },
    { "path": "registry/new-york/hello-world/use-hello.ts",    "type": "registry:hook" },
    { "path": "registry/new-york/hello-world/format.ts",       "type": "registry:lib" },
    { "path": "registry/new-york/hello-world/page.tsx",         "type": "registry:page", "target": "app/hello/page.tsx" }
  ],
  "cssVars": {
    "theme": { "font-heading": "Poppins, sans-serif" },
    "light": { "brand": "oklch(0.62 0.18 250)" },
    "dark":  { "brand": "oklch(0.72 0.16 250)" }
  }
}
```

**Fields**

| Field | Purpose |
|---|---|
| `name` | Unique slug; consumers install it as `@tekrogen/<name>`. |
| `type` | The item kind (table below). |
| `title` / `description` | Human-readable metadata. |
| `files[]` | Source files, each with a `path` and per-file `type`. Add `target` to force where it lands in the consumer (required for `registry:page`/`registry:file`). |
| `dependencies` | npm packages to install (`pkg@version` pins allowed). |
| `devDependencies` | npm dev packages. |
| `registryDependencies` | Other registry items to pull in first — bare names (shadcn core, e.g. `button`), namespaced (`@tekrogen/x`), or full URLs. |
| `cssVars` | Theme tokens injected into the consumer's CSS (`theme` / `light` / `dark`). |

**Item / file `type` values**

| Type | Use for |
|---|---|
| `registry:component` | A standalone component. |
| `registry:block` | A multi-file feature (block) — often the top-level `type`. |
| `registry:ui` | A primitive UI component (button, input, …). |
| `registry:hook` | A React hook. |
| `registry:lib` | A utility/lib module. |
| `registry:page` | A route/page — **requires** `target`. |
| `registry:file` | An arbitrary file — **requires** `target` (e.g. `~/hello.config.ts`). |

## 4. Build

Compile the manifest into per-item JSON:

```bash
pnpm registry:build          # = pnpm dlx shadcn@latest build
```

Output defaults to **`public/r/<item>.json`** — one file per item. In this repo
`public/r/` is **git-ignored**: the `dev`/`build` scripts run `registry:build`
first, so Vercel regenerates the JSON on every deploy. Don't commit it. Re-run
locally after any change to source files or `registry.json`.

## 5. Host

Serve `public/r/` over HTTPS so the URL matches the `components.json` template:

```
https://<your-domain>/r/<item>.json
```

On Vercel (the registry-starter target), `public/` is served as-is — no extra
config. For a **private** registry, gate `/r/*` behind auth in `middleware.ts`
using the `REGISTRY_AUTH_TOKEN` env var (see `env.example`); consumers then pass
a token. Local dev: `http://localhost:3000/r/{name}.json`.

## 6. Consume (install items in another project)

In the consuming project, either configure the namespace once in its
`components.json` `registries` block (Step 1) and install by name:

```bash
pnpm dlx shadcn@latest add @tekrogen/login-form
pnpm dlx shadcn@latest add @tekrogen/header @tekrogen/footer   # multiple
```

…or register an ad-hoc namespace inline (handy for local testing):

```bash
pnpm dlx shadcn@latest registry add @tekrogen=http://localhost:3000/r/{name}.json
pnpm dlx shadcn@latest add @tekrogen/login-form
```

`shadcn` resolves `registryDependencies` transitively, installs npm
`dependencies`, writes files to their `aliases`/`target` paths, and merges
`cssVars` into the consumer's theme.

## Verify

```bash
# manifest is valid JSON
pnpm dlx shadcn@latest build && ls public/r

# an item resolves end-to-end (serve first, then in a scratch project)
pnpm dlx shadcn@latest add @tekrogen/login-form --dry-run   # preview without writing
```

Confirm: each `items[]` entry produced a `public/r/<name>.json`; the served URL
returns that JSON; and an `add` copies the files + deps into a consumer.

## Gotchas

- **`pnpm dlx`, never `npx`** — `npx` aborts with `EBADDEVENGINES` under this
  repo's pnpm pin.
- **Source files must live under `registry/<style>/`** and match the `style` in
  `components.json`, or the build can't find them.
- **`registry:page` / `registry:file` need `target`** — without it the build
  errors or the file has nowhere to land.
- **`registryDependencies` are item names, not file paths** — bare for shadcn
  core, `@tekrogen/x` for our own, full URL for third-party.
- **Rebuild after every edit** — `public/r/*.json` is a build artifact; stale
  JSON ships stale components.
- **Changes here follow the repo conventions** — Conventional Commits (no emoji),
  branch `<type>/<issue#>-<slug>`; registry changes ship via the normal
  release-please flow (see README → Release Management).
